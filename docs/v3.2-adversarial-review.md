# v3.2 Adversarial Review

Review date: 2026-05-08

## Review Scope

This review only targets v3.2 behavior in:

- `docs/etf-semantic-query-spec-v3.md`
- `docs/v3-coverage-matrix.md`
- `etf-query-test-questions.md`

The review standard is strict:

1. v3.2 must implement semantic input to query execution through Text-to-Query AST.
2. Passing behavior must not rely on prepackaged business functions, deterministic AST builders, intent-dispatch templates, or local field-list assemblers as the normal path.
3. For executable v3.2 scope, LLM output must be a complete constrained AST Draft; validator/compiler may normalize and enforce safety but must not silently create the semantic query.
4. The PM test questions must be handled according to explicit v3.2 capability boundaries without quiet degradation, partial substitute answers, or test-specific hardcoding.

## Main Review Decision

The original findings below remain as concrete evidence and adversarial test ideas. They are not a flat backlog. The release-blocking issue is whether v3.2 can prove that executable answers are produced by semantic input -> LLM constrained AST Draft -> validator -> compiler -> query execution, rather than by local templates, deterministic builders, hidden fallback, or business-function dispatch.

For v3.2 strict pass, the highest-priority rule is:

> Every user-requested semantic `select`, `where`, `order_by`, `sub_intent`, period-specific field, and user-requested limit in `validated_ast` must originate from `llm_draft_ast`. Validator, compiler, orchestrator, formatter, catalog, and Registry code may constrain, normalize, audit, and physically compile that draft; they must not create the semantic query.

## P0 Meta Findings

### MF-1. Strict semantic provenance invariant is missing

v3.2 cannot be accepted as Text2SQL if the final validated AST is correct but the semantic clauses were completed by validator baselines, field profiles, period mappers, routing signals, catalogs, orchestrators, or formatters.

Required fix:

- Add a `draft_ast -> validated_ast -> compiled_query` provenance diff to every strict-pass executable case.
- Treat `select`, `where.field`, `where.op`, `order_by`, `sub_intents`, period-specific fields, and user-requested limits as semantic structure.
- Reject strict pass when semantic structure appears in `validated_ast` but is absent from `llm_draft_ast`.
- Allow only identity/context/display fields, value normalization, catalog-normalized entity values, and non-semantic execution caps as validator additions, and classify each addition.
- Add mutation tests that remove requested fields, predicates, sorts, period fields, or sub-intents from the LLM draft and prove validator rejects rather than repairs.

Covered findings: #2, #3, #6, #7, #8, #9, #10, #25, #31, #32, #43, #44, #52, #55, #56, #57, #69, #97, #117, #118, #119.

### MF-2. Non-strict outcomes are counted as Text2SQL success

Blocked, fallback, dry-run, mock-gate, partial-composite, and pre-prompt unsupported paths may be useful product behavior, but they are not question-level v3.2 Text2SQL success.

Required fix:

- Report fallback health separately from strict executable pass.
- Do not count manager-detail fallback, blocked report partials, mock gates, dry-run paths, or deterministic fallback as strict pass.
- Require `ast_generation_mode=llm_ast_draft` for every strict executable success.
- For mixed/composite questions, choose one explicit policy: whole-query unsupported, explicit partial composite, or full executable composite. Ambiguity blocks release.

Covered findings: #1, #20, #39, #40, #46, #51, #91, #120.

### MF-3. Anti-hardcoding and generalization gates are insufficient

Exact PM-question pass can be achieved by keyword classifiers, fixed JSON, or deterministic templates. It is necessary smoke evidence, but not sufficient release evidence.

Required fix:

- Require 100% canonical smoke pass for each v3.2 executable capability.
- Use 90% thresholds only for paraphrase/fuzz robustness, never for canonical capability correctness.
- Add seed-free paraphrases, counterfactuals, mutation tests, and held-out wording for every v3.2 profile and filter/date capability.
- For compare/filter/search, prove that changing requested dimensions changes `select`, `where`, or `order_by`; equivalent paraphrases should remain semantically equivalent.
- Compare semantic AST equivalence, not exact raw LLM JSON text.

Covered findings: #5, #12, #14, #24, #30, #100, #105, #106, #108, #116, #118.

### MF-4. Runtime auditability and reproducibility are not strong enough

Without full prompt, schema, Registry, generation context, model, raw draft, retry, and remote execution provenance, a passing result cannot prove it came from constrained LLM AST generation.

Required fix:

- Record model id, decoding parameters, prompt template version/hash, AST schema version, Registry version/hash, generation context hash, gate/catalog snapshot hash, raw draft artifact hash, retry transcript, and validator addition classification.
- Store full raw LLM drafts and failed draft attempts in sidecar artifacts keyed by `run_id`; response JSON may contain previews only when sidecar path/hash is present.
- Record `remote_query_executed`, query params hash, row count, remote error, and remote snapshot/as-of metadata.

Covered findings: #38, #45, #58, #101, #102, #103, #104.

### MF-5. Scope ambiguity is a release blocker

v3.2 cannot leave important PM composites in "should likely" or optional status. Optionality lets a weaker implementation avoid meaningful Text2SQL cases while still claiming compliance.

Required fix:

- For each PM composite, declare exactly one mandatory outcome in the coverage matrix: strict executable, explicit partial composite, or non-AST unsupported.
- If executable, require merged or child LLM AST Draft provenance.
- If unsupported, require a specific reason and no substitute partial answer counted as pass.
- Split coverage phase columns into first-supported phase, current-release required, and legacy-migration required.

Covered findings: #21, #22, #23, #48, #120.

### MF-6. Compiler is safety and physical planning only

The compiler may turn validated AST semantics into safe physical Mongo operations, but it must not introduce new business semantics.

Required fix:

- Permit only allowlisted physical expansions such as `__search_text__` to controlled search OR, latest-value extraction, projection shaping, read-only params, and bounded aggregation.
- Record every compiler expansion and ensure it is derived from an AST semantic operator already present in the draft.
- Forbid hidden filters, hidden sort semantics, extra metrics, arbitrary OR, post-filter/post-sort after limit, write stages, server-side JavaScript, and business intent templates.

Covered findings: #13, #73, #74, #84, #109, #110, #111, #114.

## Semantic Source Policy

| Component | May provide | Must not provide |
| --- | --- | --- |
| LLM Draft | Requested semantic fields, predicates, operators, sort, sub-intents, period-specific fields, requested limit, composite child semantics | Executable SQL/PyMongo, final answer text |
| Validator | Schema validation, field allowlist checks, value normalization, catalog-normalized entity values, identity/context/display-only additions, non-semantic execution cap | Missing requested field/op/order/sub-intent repair, semantic baseline completion, field/op override counted as success |
| Orchestrator | Evidence and values from previous steps, child-query coordination, provenance stitching | Child AST field/op templates, injected semantic clauses, substitute partial success unless explicitly marked |
| Compiler | Safe physical plan, projection shaping, allowlisted search OR, latest-value extraction, bounded aggregation | Hidden filters, hidden sort, extra requested metrics, arbitrary OR, business-function execution |
| Formatter | Registry labels, display ordering, truncation disclosure, answer rendering from returned rows | Query repair, adding unqueried semantic fields, hiding missing requested semantics |

## Strict Pass Acceptance Criteria

1. `ast_generation_mode=llm_ast_draft`.
2. Each executable strict pass emits `draft_ast`, `validated_ast`, `compiled_query`, and a provenance diff.
3. Any semantic field, operator, predicate, sort, sub-intent, period field, or requested limit in `validated_ast` that is absent from `llm_draft_ast` fails strict pass.
4. Validator negative-contract tests prove missing requested semantics are rejected, not repaired.
5. Poisoned `routing_signals.legacy_filters`, `legacy_sort_hint`, and `legacy_limit_hint` do not affect compiled query output.
6. Covered executable paths do not call legacy intent dispatch, prepackaged business functions, or deterministic AST builders as query construction sources.
7. Canonical v3.2 capability smoke is 100%; robustness thresholds are reported separately.
8. Compare, filter, and search counterfactual tests prove different user semantics produce different AST semantics.
9. Blocked, fallback, mock, dry-run, unsupported, and partial answers are separately bucketed and excluded from strict pass.
10. Runtime reports include model, prompt, schema, Registry, generation context, gate/catalog, raw draft, retry, validator-addition, compiler-expansion, and remote-execution audit fields.
11. Formatter output is derived from compiled projection and returned rows; formatter templates cannot hide missing query semantics.
12. Composite queries preserve child or merged AST provenance; cross-step handoff may pass evidence and values, not field/op templates.

## Findings

### 1. `manager_detail` blocked fallback is counted as executable success

`manager_detail` is blocked unless the verification gate passes, but the coverage matrix marks manager-detail questions as `ExecutableQuery / single / manager / llm_ast_draft / remote_query_allowed=true`.

This turns "user asked for tenure/history/managed scale" into "query current manager" and risks counting a substitute answer as v3.2 Text2SQL success.

Recommendation:

- Represent blocked manager-detail questions as `UnsupportedQuery`, `BlockedWithFallback`, or equivalent non-success outcome.
- If a manager fallback is returned, count it only as fallback health, not as question-level Text2SQL success.
- Set `ast_required=false` and `remote_query_allowed=false` for the original blocked capability, or add explicit fallback columns that do not count toward strict pass.

### 2. Compare fixed 8 columns are a prepackaged template

Compare ignores user-specified dimensions and always shows a fixed 8-column template.

This conflicts with semantic query behavior for questions like "compare scale and fees" or "compare returns".

Recommendation:

- Let LLM select requested compare fields.
- Validator may add identity/context fields.
- Keep the fixed 8-column template only for generic compare without specified dimensions.

### 3. Validator baseline completion can become query construction

The spec allows validator to补齐 baseline `answer_fields`, reorder fields, and preserve legacy answer shape.

If baseline contains semantic fields, LLM is no longer the source of the query structure.

Recommendation:

- Validator may add identity/context fields only.
- User-requested semantic fields must be present in the LLM Draft.
- If semantic fields are missing, fail validation rather than silently completing them.

### 4. Evidence builder `candidate_fields` can become field selection

Evidence builders can map phrases directly to candidate fields. This is useful, but if only one candidate is supplied and tests only check final AST, the local builder effectively performed field selection.

Recommendation:

- Treat `candidate_fields` as evidence, not as a field whitelist or final answer.
- Use multiple-candidate disambiguation tests.
- Verify every LLM-selected semantic field against user evidence, Registry allowance, or explicit clarification.
- Add tests where local phrase candidates are incomplete or over-specific, and the strict-pass result still depends on LLM Draft semantics rather than the evidence builder's final field choice.

### 5. Coverage matrix can encourage hardcoding PM originals

The PM matrix is built from exact questions. Without paraphrase/fuzz gates, a deterministic mapper can pass.

Recommendation:

- Add paraphrase sets for every v3.2 capability.
- Require equivalent AST semantics across paraphrases.
- Report PM exact pass separately from paraphrase/fuzz pass.

### 6. `routing_signals` can carry complete legacy filters

`routing_signals` may contain `legacy_filters`, `legacy_sort_hint`, and `legacy_limit_hint`. Even if not sent to LLM, validator or orchestrator could reuse them.

Recommendation:

- Do not include legacy completed conditions in the passing runtime object.
- Store them only in separate debug artifacts.
- Add tests proving passing paths do not read `legacy_filters`, `legacy_sort_hint`, or `legacy_limit_hint`.

### 7. Orchestrator can inject semantics into child queries

The spec says orchestrator may coordinate composites, but also mentions injected `in` behavior. This risks child ASTs being built outside LLM Draft.

Recommendation:

- Orchestrator may pass only evidence, such as Step 1 fundcodes.
- Step 2 LLM must output a complete child AST Draft.
- Validator may only verify or override values, never field/op structure.

### 8. Period handling can remain a local classifier plus field mapper

Period fallback can normalize period and map it through local `PERIOD_FIELDS`. If LLM only chooses period and local code chooses fields, performance is not full AST generation.

Recommendation:

- Period normalization should produce evidence only.
- LLM Draft must explicitly select period-specific fields.
- Validator should verify field-period consistency, not generate the core performance field set.

### 9. Default sorting in validator can alter query semantics

Validator may add default `order_by`, for example scale descending.

Recommendation:

- Separate `order_by` from `display_order`.
- Only user-expressed sort semantics should become `order_by`.
- Product display defaults must be marked separately and excluded from query semantics.

### 10. `selection_context.field_profile` remains a strong template

Profiles restrict fields and baselines so tightly that LLM may only fill a template.

Recommendation:

- Require same-profile questions to produce different `select` sets.
- Require evidence-to-field alignment checks.
- Track whether semantic fields were selected by LLM or added by validator.

### 11. Coverage column `must_migrate_in_v3_2` is ambiguous for v3.2 additions

v3.2-added executable rows have `must_migrate_in_v3_2=false`, which can be read as "not required".

Recommendation:

- Rename to `must_migrate_legacy_in_v3_2`.
- Add `required_for_v3_2_release`.
- Set v3.2 executable additions to `required_for_v3_2_release=true`.

### 12. v3.2 golden threshold permits failures in new capabilities

The spec allows 90%+ for the v3.2 layer golden set, which may hide a broken capability.

Recommendation:

- Require 100% smoke for each v3.2 capability.
- Allow 90% only for paraphrase/fuzz robustness, not canonical AST correctness.

### 13. Search compiler expands hidden OR

`__search_text__ contains` compiles to controlled multi-field OR while AST does not expose OR.

Recommendation:

- Record `compiler_expansion=search_text_or`.
- Include expanded fields in `mongo_params`.
- Add a guard preventing arbitrary future OR injection in compiler.

### 14. v3.2 added tests are too narrow

The PM set has very few v3.2 examples for base extensions, date filters, and composite_single.

Recommendation:

- Add paraphrases for each v3.2 field.
- Add date eq/gte/lte/sort tests.
- Add multiple composite_single variants.

### 15. PM "basic information" and v3 baseline conflict

PM says "全部基础字段" for basic info, while v3.2 keeps the v3.0 four-field baseline.

Recommendation:

- Clarify that `basic_info` is a legacy baseline query.
- Add a distinct v3.2 "full basic information" case or return clarification.

### 16. `investment_profile` field selection is under-tested

"投资目标和策略" should select only objective and strategy fields, not the full investment profile.

Recommendation:

- Golden AST must assert exact selected fields.
- Add negative checks for unrequested long-text fields.

### 17. PM bucket and runtime profile diverge for risk-return features

PM groups "风险收益特征" under basic information, while runtime should route it to `investment_profile`.

Recommendation:

- Add an explicit test proving PM bucket is not used for runtime dispatch.

### 18. Date filter support is broader than tests

Spec claims date `eq/gte/lte/between` and sort support, but PM covers only year-between.

Recommendation:

- Add tests for exact date, after date, before date, earliest, and latest.

### 19. Subscription status vs investment advice needs more boundary tests

"能申购吗" is executable; "能买吗" is deny. The current tests do not sufficiently stress this boundary.

Recommendation:

- Add executable tests for subscribe/redeem phrasing.
- Add deny tests for buy/invest phrasing.

### 20. Search-plus-report composite has conflicting partial behavior

One section suggests partial search may execute before report gate, while coverage marks the whole question unsupported.

Recommendation:

- Choose one rule.
- Prefer whole-query unsupported unless coverage explicitly marks `partial_composite=true`.

### 21. PM 十.1 has no mandatory v3.2 outcome despite matching filter -> detail

"Find the lowest-fee HS300 ETF, then show basic info and returns" is a core Text2SQL composite, but coverage marks it later phase.

Recommendation:

- Choose one mandatory v3.2.0 outcome in the coverage matrix: strict executable two-step Text2SQL, explicit partial composite, or non-AST unsupported.
- If unsupported, explicitly narrow `filter -> single detail` scope and require no substitute partial answer to count as pass.

### 22. PM 十.5 needs a mandatory `composite_single` decision

"Scale, fee, return" for one fund is same fundcode, same collection, multiple single profiles. Coverage marks it unsupported.

Recommendation:

- Either include it as strict `composite_single`, or define a hard unsupported boundary such as max two sub-intents and explain why.
- In either case, make the coverage row mandatory and testable rather than optional.

### 23. PM 十.7 can be swallowed by local period/template logic

"Since inception return and dividends" must be a real `composite_single` AST, not local period mapping plus dividend template.

Recommendation:

- Golden AST must assert `intent=composite_single`, `sub_intents`, `ths_yeild_std_fund`, dividend fields, and fundcode where.

### 24. Search tests overuse explicit trigger words

Current search PM cases mostly contain "search/find/name contains".

Recommendation:

- Add semantic search phrasing without explicit search verbs.
- Add search/filter boundary tests for tracking-index phrases.

### 25. Compare specified dimensions are weakened in coverage

PM asks to compare scale/fee/return, while coverage treats all as generic compare.

Recommendation:

- Require field-specific compare AST in v3.2, or mark fixed-column compare as legacy baseline rather than strict Text2SQL.

### 26. Existence check is forced through basic-info template

`000001有这只ETF吗` is marked `basic_info`, encouraging a full basic info query for an existence check.

Recommendation:

- Add `existence_check` or allow minimal `select=["fundcode"]`.

### 27. Name resolution cases are under-tested

The PM set tests invalid non-code text but not ambiguous or unique ETF names as executable semantic input.

Recommendation:

- Add unique name, ambiguous name, and unresolved name cases.

### 28. Registry vs manually maintained views weakens spec reviewability

The spec says runtime follows Registry, while views may remain manually maintained in v3.2.

Recommendation:

- Generate and persist a v3.2 Registry snapshot for review.
- Compare coverage expected profile against enabled Registry capabilities in tests.

### 29. Deny and unsupported boundaries are too sparse

Several v3.2-adjacent deny boundaries are missing, especially objective ranking vs investment advice.

Recommendation:

- Add paired executable/deny examples for "fee lower" vs "better to buy", "can subscribe" vs "can buy", and "highest return" vs "recommend".

### 30. Golden `expected LLM AST Draft` can encourage fixed JSON

Exact draft matching may overfit model output.

Recommendation:

- Compare semantic AST equivalence, not raw JSON text.
- Keep raw LLM output for audit only.

### 31. Answer shape regression can rewrite LLM AST

Validator may repair fields to maintain old answer shape, diluting LLM-generated query structure.

Recommendation:

- Reject semantic-field omissions instead of repairing them.
- Only identity/context additions may remain strict-pass compatible.

### 32. "Registry generates answer_fields" conflicts with v3.2 LLM Draft

This wording points back to deterministic answer template generation.

Recommendation:

- Registry should provide allowed/required field policies, not generate final answer_fields in passing paths.

### 33. Search fixed list baseline is not full Text2SQL field selection

Search output is fixed list columns.

Recommendation:

- Report search as where-generation Text2SQL with baseline display template.
- Do not count it the same as full semantic field selection.

### 34. Filter baseline can hide missing requested fields

Filter default columns may omit the actual filter metric, such as date or return.

Recommendation:

- Where/order_by metric fields should be displayed by default unless explicitly hidden.

### 35. Filter default period `1y` is a semantic assumption

For "highest return", defaulting to one year can be wrong.

Recommendation:

- Require clarification for unspecified return periods in filter/sort, or visibly mark default period usage.

### 36. Default `order_by` should not be query semantics

System display ordering is not the same as user-requested order.

Recommendation:

- Split `order_by` and `display_order`.

### 37. `select` and `answer_fields` need role constraints

The spec does not explain why a selected field may be non-displayed or why a requested field may be omitted.

Recommendation:

- Add field roles: `display`, `filter`, `sort`, `identity`, `support`.

### 38. Runtime output lacks generation context audit

Without context snapshot/hash, a result cannot prove that LLM stayed inside v3.2 constraints.

Recommendation:

- Output `generation_context_hash`, `selection_context_snapshot`, `registry_version`, and `gate_snapshot`.

### 39. Manager fallback should not count as question pass

Repeated spec sections allow manager fallback to count as manager-path success.

Recommendation:

- Count fallback separately; do not count the original manager-detail question as passed.

### 40. Report composite partial behavior conflicts

One section says base can execute when report is blocked; another says whole composite unsupported unless explicitly partial.

Recommendation:

- Prefer explicit partial-only behavior and remove conflicting language.

### 41. `all` -> limit 50 distorts user intent

"All" is converted to capped limit.

Recommendation:

- Represent requested all plus execution cap separately and disclose truncation.

### 42. Search keyword stripping is fragile

Simple stopword deletion can damage valid product/index names.

Recommendation:

- Use span-based extraction and normalize with audit fields.

### 43. Top-level flow says "condition parsing" before LLM

This wording can justify local where generation.

Recommendation:

- Rename to "condition evidence extraction".

### 44. Validator correction is not quantitatively bounded

The spec does not define how much validator-added semantics is too much.

Recommendation:

- Add strict pass metrics for LLM-selected semantic fields and validator-added semantic fields.

### 45. Runtime output cannot prove constrained generation

The required audit fields are insufficient.

Recommendation:

- Include evidence/context/gate/registry snapshots or hashes in every v3.2 report.

### 46. Mock gates can contaminate release validation

Dry-run mock gates are allowed but not excluded from release pass.

Recommendation:

- Disallow `dry_run_gate=true` in release/E2E pass statistics.

### 47. Gate state lacks versioned evidence

Gate files need verification metadata.

Recommendation:

- Record script version, sample size, collection, field, generated_at, and data snapshot metadata.

### 48. Coverage phase columns are overloaded

`release_scope`, `executable_in_current_phase`, and `must_migrate_in_v3_2` mix separate concepts.

Recommendation:

- Split first-supported phase, required-in-current-release, and legacy-migration-required.

### 49. `v3_ast` name is ambiguous

It may mean draft, validated, or legacy AST.

Recommendation:

- Rename to `draft_ast`, `validated_ast`, and `compiled_mongo_params`.

### 50. Failure schema differs from normal runtime schema

Failures use a nested `v3` object while normal output uses top-level fields.

Recommendation:

- Use one schema for success and failure.

### 51. `failure_stage=gate` is ambiguous

Gate-blocked capabilities should generally not enter LLM Draft at all.

Recommendation:

- Pre-prompt gate block should be non-AST unsupported.
- Blocked fields emitted by LLM should be validator failures.

### 52. Deterministic value overrides need strict audit

Validator may override values safely, but this can hide LLM errors.

Recommendation:

- Output path-level override records.
- Never allow field/op override as success.

### 53. `raw_value` is not fully schema-defined

Validated where items should always preserve raw source when normalization occurs.

Recommendation:

- Add `raw_value`, `normalized_from`, and `entity_source` schema requirements.

### 54. Generated views conflict with v3.2 manual maintenance

One section delays generated full views to v3.3; another says all views are generated from Registry.

Recommendation:

- Define a minimal generated-view set required in v3.2.

### 55. `period=all` validator expansion is local template generation

Validator may补齐 all-period fields.

Recommendation:

- LLM must output all selected period fields; validator only verifies set equality.

### 56. `baseline_fields_added` lacks semantic classification

The list does not distinguish identity fields from semantic fields.

Recommendation:

- Split identity/context/semantic additions and exclude semantic additions from strict pass.

### 57. Coverage does not constrain validator additions

Tests do not specify allowed defaults or added fields.

Recommendation:

- Add coverage columns for allowed defaults and allowed baseline additions.

### 58. `remote_query_allowed` does not prove remote was not called

Expected allowance is not actual execution audit.

Recommendation:

- Add `remote_query_executed`, rows returned, remote error, and params hash.

### 59. LLM-generated labels can affect final answer

Labels are user-visible and not required to be canonical.

Recommendation:

- Final labels should come from Registry, not LLM.

### 60. JSON schema does not forbid extra fields

LLM can output unused or dangerous extra keys.

Recommendation:

- Use strict JSON schema with `additionalProperties=false`.

### 61. Prompt injection is not explicitly handled

The prompt does not say user question is untrusted.

Recommendation:

- Add prompt and validator rules for injection-like user text.

### 62. Existence queries are forced to have answer fields

Non-compare answer_fields cannot be empty, forcing template-style output.

Recommendation:

- Add `exists` output style or minimal existence projection.

### 63. `output_style` needs intent/profile-level validation

It should equal the Registry output style for the chosen capability.

Recommendation:

- Validate profile-level output style for single-profile queries and composite-policy-level output style for composite queries.
- Reject output styles that let a broad query mode hide a mismatched capability.

### 64. Empty `where` policy is undefined by mode

Some filters can have empty where and only sort, but single/search/compare cannot.

Recommendation:

- Add per-mode `where_cardinality_policy`.

### 65. `order_by` schema is underspecified

Direction and object shape are not formally defined.

Recommendation:

- Define order_by schema and enum.

### 66. Draft `where.value` shapes are inconsistent

Some examples use strings, others `{raw: ...}`.

Recommendation:

- Define operator-specific Draft and Validated value schemas.

### 67. Duplicate fields are not forbidden

`select` and `answer_fields` uniqueness is not specified.

Recommendation:

- Reject duplicate select and answer field entries.

### 68. Non-report `report_period` / `expand` null rule is missing

Recommendation:

- Require both to be null for non-report modes.

### 69. `sub_intents` can include unrequested single intents

Recommendation:

- Require every emitted `sub_intent` to be evidence-backed by the user question or explicit clarification state.
- Reject validator/orchestrator additions of unrequested sub-intents.
- Do not require a mechanical exact match to local `sub_intent_candidates`; candidates are evidence, not the final semantic source.

### 70. Parser leniency for JSON is unspecified

Recommendation:

- Strict pass should accept only a single JSON object, not extracted fenced JSON plus prose.

### 71. `select` vs where/order_by display semantics are unclear

Recommendation:

- Requested metric fields should appear in answer_fields by default.

### 72. Missing adversarial schema tests

Recommendation:

- Add tests for unknown fields, extra keys, OR, regex, invalid labels, over-limit, report fields in non-report, and malformed JSON.

### 73. latest_ts array filter/sort semantics are underspecified

Recommendation:

- Define aggregation or latest-value extraction before filter/sort/limit.

### 74. Expected Mongo params cannot represent aggregation

Recommendation:

- Add `operation`, `pipeline`, and `computed_fields`.

### 75. Date gate verifies format but not comparison semantics

Recommendation:

- Verify raw Mongo date/string ordering semantics and dirty values.

### 76. Numeric value ranges are too coarse

Recommendation:

- Move to field-specific `value_schema`.

### 77. Percent storage scale needs field-level verification

Recommendation:

- Registry should record `storage_scale` and formatter/normalizer scale.

### 78. Ranking sort null handling is undefined

Recommendation:

- Sort ranking numeric fields only; nulls last.

### 79. latest `btime` handling lacks timestamp policy

Recommendation:

- Define btime format, timezone, as-of display, and cross-field date consistency.

### 80. null semantics miss dirty values

Recommendation:

- Field gate should store verified empty-value set.

### 81. Search regex normalization is insufficient

Recommendation:

- Define case, full-width/half-width, whitespace, and alphanumeric normalization.

### 82. Remote-data regression policy is too weak

Recommendation:

- Validate predicate truth, sort monotonicity, and top-N fundcodes, not just format.

### 83. Limit truncation lacks total/has_more semantics

Recommendation:

- Add `has_more`, `limit_applied`, and truncation reason.

### 84. Post-filter/post-sort after limit is not forbidden

Recommendation:

- Require all semantic filtering and sorting before limit.

### 85. Clarification candidates require remote data but non-AST says no remote access

Clarification rules say `ClarificationRequired` does not call remote Mongo, but also require fund candidates to be sorted by scale. Sorting by scale generally requires catalog data from the remote base collection.

Recommendation:

- Define whether clarification may use a prebuilt local catalog snapshot.
- Add `catalog_snapshot_used` to runtime audit.
- Do not perform live remote Mongo calls in non-AST clarification paths.

### 86. Clarification truncates candidates and loses recoverability

The spec keeps only the top five options and excludes truncated candidates from `state_id`. This can make a valid user choice impossible if it was ranked sixth.

Recommendation:

- Keep the full candidate set server-side under `state_id`, or provide pagination.
- Do not discard candidates needed for follow-up disambiguation.

### 87. Clarification reason enum is too narrow

The schema only lists `name_ambiguity`, `insufficient_conditions`, and `too_many_candidates`, while later sections use `period_ambiguity`, `field_ambiguity`, `value_ambiguity`, and `missing_search_keyword`.

Recommendation:

- Expand the `ClarificationRequired.reason` enum to all used reasons.
- Add schema tests proving every emitted reason is valid.

### 88. Name resolution is critical but underspecified for v3.2

The spec relies on fundcode or "unique name" to route single and composite_single queries, but does not define the name resolver's data source, freshness, ambiguity threshold, or audit output.

Recommendation:

- Specify `FundNameCatalog` source, refresh policy, matching algorithm, ambiguity threshold, and snapshot metadata.
- Include unique-name and ambiguous-name PM/paraphrase tests.

### 89. `IndexNameCatalog` TTL and stale-cache behavior are undefined

The catalog is refreshed by TTL or manually, but the TTL value, stale behavior, and release-test snapshot are not specified.

Recommendation:

- Define TTL, stale-while-revalidate policy, and failure mode.
- Record catalog version/hash in runtime output and test reports.

### 90. Index substring matching can produce multiple valid candidates

The spec says substring match maps user text like `沪深300` to `沪深300指数`, but does not define what happens when multiple index names contain the substring.

Recommendation:

- Return `ClarificationRequired(value_ambiguity)` for multi-match cases unless one exact/alias match dominates.
- Add tests for broad terms like "红利", "消费", "医药", and "300".

### 91. Routing priority puts report before search/filter and may over-block mixed queries

Report is priority 3 while search/filter is priority 4. A query containing a report word plus executable base/filter semantics can become report-blocked before the executable part is considered.

Recommendation:

- For mixed queries, preserve all candidate intents and let composite policy decide.
- Do not let a blocked report keyword erase an independently executable v3.2 base/filter query unless coverage explicitly requires whole-query unsupported.

### 92. Single-vs-search routing relies on "unique name" but tests do not cover it

The rule suppresses search triggers when a name resolves uniquely to a fundcode, but the PM tests mostly use numeric codes or explicit search phrases.

Recommendation:

- Add tests like "帮我找工银沪深300ETF的费率" and "沪深300ETF的费率是多少" with unique/ambiguous outcomes.
- Audit whether routing used fundcode, unique name, or unresolved text.

### 93. Base-extension fields without fundcode are routed to filter, but filter semantics are unclear

The spec says listing/benchmark/subscription/linked-fund phrases without fundcode/name route to filter. For questions like "业绩比较基准有哪些", the intended where/order/limit is unclear.

Recommendation:

- Distinguish list-enumeration questions from filter queries.
- Return clarification or define an explicit aggregation/list-distinct capability; do not force them into ordinary filter AST.

### 94. `has_more=true` with no hidden candidate state blocks natural follow-up

The spec exposes `has_more=true` but says truncated candidates are not retained in `state_id`, so the user cannot ask for more or pick a hidden candidate.

Recommendation:

- Either remove `has_more` or support pagination / hidden candidate retention.

### 95. Non-AST clarification options can include `free_text`, but retry semantics are not specified

`options[].kind` allows `free_text`, but there is no contract for how free-text follow-up is merged into the prior query.

Recommendation:

- Define follow-up state merge rules.
- Ensure follow-up does not bypass v3.2 routing, LLM AST Draft, validator, or gate checks.

### 96. `recognized_query_mode=null` for Clarification loses useful route audit

Clarification for a search or single query sets `recognized_query_mode=null` by the non-AST rule, losing the route that caused the clarification.

Recommendation:

- Keep `recognized_query_mode=null` if required, but add `clarification_context.query_mode_candidate` and `intent_candidates`.
- Use that context in tests and follow-up handling.

### 97. Local catalogs can become hidden deterministic query sources

FundNameCatalog and IndexNameCatalog are allowed to provide normalized values, but without strict audit they can effectively choose the final where condition.

Recommendation:

- Treat catalog outputs as evidence with source and confidence.
- LLM must still output the field/op structure.
- Validator may only override value and must record the override source.

### 98. Catalog unavailable behavior can reduce production reliability without test coverage

IndexNameCatalog unavailable returns unsupported/clarify, but no PM test covers catalog outage or stale catalog behavior.

Recommendation:

- Add fault-injection tests for missing catalog, stale catalog, and multi-match catalog.

### 99. Coverage matrix lacks follow-up-turn tests

Clarification uses `state_id` and option ids, but tests only cover single-turn questions.

Recommendation:

- Add multi-turn tests for selecting a fund candidate and retrying through v3.2 AST generation.
- Verify the follow-up AST still has `ast_generation_mode=llm_ast_draft`.

### 100. Canonical seed routing can become a keyword classifier

The spec relies heavily on canonical seeds for `intent_candidates`. Without semantic/paraphrase tests, implementation can pass as a keyword classifier.

Recommendation:

- Require seed-free paraphrase, counterfactual, mutation, and held-out wording tests for v3.2 fields.
- Do not prescribe embeddings as the only implementation; the gate is semantic generalization beyond canonical seed words.

### 101. LLM runtime parameters are not pinned

The spec requires LLM AST Draft but does not define model id, decoding parameters, JSON/structured-output mode, max output, timeout, retry count, or fallback behavior. Two implementations can both claim v3.2 compliance while producing materially different ASTs.

Recommendation:

- Pin `model_id`, `temperature`, `top_p`, JSON/structured-output mode, timeout, retry count, and max tokens for v3.2 evaluation.
- Record these values in every runtime result and test report.

### 102. Prompt/version identity is not auditable enough

The runtime output keeps `llm_ast_draft_raw`, but not prompt template version, prompt hash, generation_context hash, Registry version/hash, or schema version. Failed or surprising ASTs will not be reproducible.

Recommendation:

- Add `prompt_template_version`, `prompt_hash`, `generation_context_hash`, `registry_hash`, and `ast_schema_version`.
- Include these in golden reports and remote smoke logs.

### 103. `llm_ast_draft_raw` may be truncated and become unusable evidence

The schema allows raw draft truncation. That weakens the core claim that an executable answer came from LLM-generated AST rather than local construction.

Recommendation:

- Store the full raw draft in a sidecar artifact keyed by `run_id`.
- Allow the response JSON to include a truncated preview only if the full artifact path/hash is present.

### 104. JSON parse failure handling can mask prompt/schema problems

The spec says parse/schema failures are `llm_ast_draft_failed`, but does not require preserving invalid raw output, parse error class, or retry transcript. A wrapper could retry until it gets valid JSON and hide systematic prompt issues.

Recommendation:

- Record every failed raw response, parse error, retry index, and final selected draft.
- Count retries separately in evaluation.

### 105. The expected LLM AST Draft oracle is brittle

Golden records require `expected LLM AST Draft`, but LLM drafts can vary while still compiling to the same validated semantics. Exact draft matching can encourage prompt overfitting or deterministic template generation.

Recommendation:

- Evaluate draft structure by semantic constraints, not exact JSON equality.
- Use exact matching only for `validated_ast` / compiled query semantics where appropriate.

### 106. CI gate commands are not concrete

The spec defines gates and metrics but does not name the command(s), fixtures, expected artifacts, or failure thresholds that release CI must run.

Recommendation:

- Define one canonical v3.2 verification command.
- Require it to emit coverage matrix alignment, AST-generation buckets, schema failures, remote smoke, and formatter regression reports.

### 107. Generated views can drift without a regeneration check

Generated views are read-only derivatives of Registry, but there is no required CI check that regenerates them and fails on diff.

Recommendation:

- Add a `registry -> generated views` reproducibility check.
- Fail CI when checked-in views diverge from Registry output.

### 108. Coverage matrix can drift from PM questions

The spec says coverage matrix must cover every PM question, but there is no mechanical check that IDs/questions match `etf-query-test-questions.md`.

Recommendation:

- Add a parser-based alignment check that verifies every PM row appears exactly once in the coverage matrix.
- Fail on missing rows, duplicate rows, renamed questions, or stale expected outcomes.

### 109. Remote execution safety is underspecified

The compiler produces Mongo params, but the spec does not require collection allowlists, read-only enforcement, banned stages, max time, projection path validation, or result size limits.

Recommendation:

- Whitelist collections and fields from Registry.
- Forbid `$out`, `$merge`, `$function`, server-side JavaScript, and arbitrary aggregation stages.
- Enforce `maxTimeMS`, hard `limit`, projection path allowlist, and read-only credentials.

### 110. Aggregation pipeline support is ambiguous but required by data shape

The spec says many base fields are time-series arrays and latest values are used for display/sort, but `mongo_params` is still shaped as `{filter, projection, sort, limit}`. Latest-value filtering/sorting often requires an aggregation pipeline or materialized latest fields.

Recommendation:

- Either require materialized latest scalar fields in `tb_ths_etf_base`, or extend compiler output to a restricted aggregation pipeline.
- Add tests proving sort/filter happens on latest values before limit.

### 111. `allowDiskUse` and large-sort behavior are not controlled

Filter/sort queries over remote Mongo can accidentally trigger expensive full-collection sorts. The spec sets no policy for indexes, `allowDiskUse`, timeout, or explain-plan checks.

Recommendation:

- Disable `allowDiskUse` by default for v3.2 smoke.
- Require index/explain validation for every sortable field in the current phase.
- Fail gate if a covered query requires unbounded full scan + sort.

### 112. Query result snapshot policy is missing for volatile fields

The test policy distinguishes stable vs volatile fields, but does not require a fixed remote snapshot or as-of timestamp for v3.2 acceptance. Latest NAV, scale, shares, and returns can move.

Recommendation:

- Use a fixed fixture snapshot for golden semantic tests.
- For live remote smoke, record `as_of`, snapshot/catalog version, and tolerance policy per field.

### 113. Field-level numeric normalization lacks negative and boundary cases

The spec mentions amount/percent/date normalization, but does not require tests for negative values, zero, inclusive boundaries, Chinese units, decimal percentages, or percent vs ratio storage.

Recommendation:

- Add adversarial normalization tests for `>`, `>=`, `<`, `<=`, `between`, and exact threshold equality.
- Require compiled query values to be asserted, not just answer format.

### 114. Limit policy may hide semantic failures

Default limit can be inserted by validator and remote answer may look plausible even if where/order_by semantics are wrong. Current gates do not require asserting the full predicate over returned rows.

Recommendation:

- For every filter/sort smoke, verify every returned row satisfies predicates.
- Verify sort monotonicity across returned rows and a top-N oracle for fixed fixtures.

### 115. Partial entity handling can weaken compare semantics

Compare allows retaining valid fundcodes when some are invalid, but PM boundary expects "部分找到". Without a strict answer contract, this can pass while silently ignoring invalid user input.

Recommendation:

- Require `partial_entity_found` output with found/missing entity lists.
- Do not present a normal compare answer without explicit missing-entity notice.

### 116. Deny vs objective ranking boundary is under-tested

The PM file distinguishes objective sorting from recommendation, but coverage can still pass with keyword deny rules. Phrases such as "哪只费率更低" and "哪个更好" need semantic boundary tests.

Recommendation:

- Add paired tests where only subjective advice is denied and objective metric comparison remains executable.
- Require deny decisions to include evidence span and deny category.

### 117. The prompt can still leak operational templates through `selection_context`

Even if filters are banned from prompt, `baseline_answer_fields`, field profiles, and examples can become a near-complete template for single-profile questions. This weakens the distinction between Text2SQL and local intent-to-template.

Recommendation:

- Minimize prompt examples that mirror PM questions.
- Require equivalent paraphrases to produce semantically equivalent ASTs.
- Require different requested dimensions to change `select`, `where`, `order_by`, or `sub_intents` rather than being absorbed into the same profile baseline.
- Require field/op/value decisions to trace to user evidence, not just profile baseline.

### 118. Field evidence checks do not prove semantic reasoning

The validator checks whether selected fields are traceable to evidence, but canonical seeds can provide the evidence. An implementation can still be a synonym matcher plus fixed AST.

Recommendation:

- Add paraphrases with no canonical seed words but clear semantic meaning.
- Require failure analysis to separate seed-hit, paraphrase-hit, and LLM-only semantic interpretation.

### 119. Mixed filter/search questions can be over-simplified

Questions like "跟踪沪深300指数、费率最低" require both text/index matching and metric sorting. The spec allows mixed LLM AST, but coverage does not force the compiler to preserve all conditions together.

Recommendation:

- Add tests where dropping either the search/index condition or the sort condition changes the expected answer.
- Assert compiled query includes all semantic clauses.

### 120. v3.2 scope still contains composite ambiguities that can be gamed

`composite_single` is supported only for same-fund same-collection profiles, but PM question 十.7 can be either supported or unsupported. This optionality lets a weaker implementation avoid a meaningful v3.2 composite case.

Recommendation:

- Decide one mandatory outcome for 十.7 in the coverage matrix for v3.2.0.
- If executable, require one merged LLM AST Draft or child AST Drafts with full composite provenance.
- If unsupported, require a specific non-AST reason and no partial answer counted as pass.

## Additional Findings: Spec Consistency Audit (2026-05-08)

These findings were identified during a full-spec consistency audit after the six prior optimizations were applied. They target internal contradictions, undefined terms, and structural defects in `docs/etf-semantic-query-spec-v3.md`.

### 121. Golden set requires both gate-passed and gate-blocked tests for establishment_date (BLOCKER)

Line 3192 requires ≥2 establishment_date filter/sort test cases (gate must pass). Line 3194 requires ≥1 "日期 filter gate 未通过" test case (gate must be blocked). The gate can only be in one state. Both requirements cannot be simultaneously satisfied regardless of verification outcome.

Recommendation:

- Make the golden set conditional on gate status:
  - If gate passes: ≥2 filter/sort tests; replace "gate 未通过" with "date range 不伪造为 search ×1".
  - If gate fails: ≥1 gate-blocked test; drop the ≥2 filter/sort requirement.

### 122. `filterable_text` type referenced but never defined (HIGH)

Line 1923 says filter `contains` exceptions require Registry to mark a text field as `filterable_text`, but the field type whitelist (line 456) only lists eight types (`identity`, `plain`, `text`, `amount`, `percent`, `date`, `nullable_ref`, `long_text`). `filterable_text` is not among them. Section 14 lists `ths_perf_comparative_benchmark_fund` only as `selectable` with no `filterable_text` flag or `contains` operator gate.

Recommendation:

- Add `filterable_text` to the field type table with allowed operators and normalizer, OR replace the term with an existing type + explicit `filter_operators.contains` gate definition.
- Document `ths_perf_comparative_benchmark_fund` `contains` operator gate in the field-level schema with the same rigor as `ths_etf_to_code_fund`.

### 123. Compare dimension handling has three conflicting rules (HIGH)

Three lines state incompatible behaviors for user-specified compare dimensions:

- Line 1152: user dimensions → LLM Draft **must** reflect requested fields in `select`/`answer_fields`.
- Line 1559: user dimension words **only** go to `raw_evidence`/`value_candidates` for audit, **do not change** compare baseline field set.
- Line 2440: user dimensions → Step 2 LLM Draft **must still** select corresponding semantic fields.

Lines 1152 and 2440 require LLM to change fields. Line 1559 prohibits it. Resolution is needed before implementation.

Recommendation:

- Option A (text2sql-aligned): Delete line 1559. Compare in v3.2 is within `llm_ast_draft` scope; LLM selects fields based on evidence including dimension words. Validator ensures identity columns are not dropped.
- Option B (conservative): Delete "必须在 select/answer_fields 中体现请求字段" from line 1152. Keep compare fixed 8 columns. Dimension words are audit-only.

### 124. Duplicate section number 6.3 (HIGH)

Line 1561 `### 6.3 Search/Filter Selection Context (v3.1+)` and line 1755 `### 6.3 v3.2 Base Extension Selection Context` share the same section number.

Recommendation:

- Rename line 1755 to `### 6.4 v3.2 Base Extension Selection Context`. Re-number downstream subsections accordingly.

### 125. Validator `display_order` vs "no default order_by" boundary is fuzzy (HIGH)

Lines 1992-2008 say compiler must not generate default sort, but validator can add `display_order` (product display sort). Line 1145 says validator cannot add "user's unexpressed order_by." For a filter query with no explicit sort, adding `display_order={"field": "ths_fund_scale_fund", "direction": "desc"}` and adding `order_by={"field": "ths_fund_scale_fund", "direction": "desc"}` are indistinguishable in compiler output. The boundary between the two rests on a naming convention, not a semantic rule.

Recommendation:

- Define `display_order` as formatter-only: it affects display row order but does NOT enter compiler's Mongo `sort`. If compiler needs `sort` (e.g., to pair with `limit`), the `order_by` must come from LLM Draft explicitly.
- Validator may inject `display_order` for formatter use only. Validator must not inject `order_by` into compiled query.

### 126. Period Normalization Fallback section (v3.1+) is confusing in a v3.2 spec (HIGH)

Lines 2153-2196 describe the v3.1 deterministic LLM-period-fallback mechanism in full detail. Although a disclaimer at line 2157 says it is not `llm_ast_draft`, the section's length and placement make it likely that implementers will treat it as a valid v3.2 implementation pattern. The spec elsewhere (line 2159) warns "v3.2 新增能力不得仿照该模式", but the detailed documentation undermines this prohibition.

Recommendation:

- Move the section to an appendix labeled "仅用于 v3.0/v3.1 遗留参考" or add a prominent marker at the section start.
- Cross-reference to "Performance Period Handling in LLM AST Draft" (line 1316) as the v3.2 replacement.

### 127. `composite_single` formatter output undefined (MEDIUM)

Section 17 defines formatter rules by `output_style` (summary, list, compare, report_list). `composite_single` uses `output_style=summary` but its `answer_fields` is a union of multiple profile baselines. The formatter section does not specify how to present fields from different profiles in a single summary output.

Recommendation:

- Add formatter rules for `composite_single`: display fields in `answer_fields` order, each with its canonical format. Long-text fields use truncation. Null fields show `暂无数据`.

### 128. `manager_detail` blocked fallback golden test requires a question that triggers `manager_detail` intent (MEDIUM)

Line 3191 requires ≥1 `manager_detail` blocked fallback test case. But if the golden set only contains questions that trigger the `manager` intent (e.g., "510300的基金经理是谁"), not questions that trigger `manager_detail` (e.g., "510300基金经理的历史业绩怎么样"), the blocked fallback path is never exercised.

Recommendation:

- Explicitly list a question that triggers `manager_detail` intent in the golden set, e.g., "510300基金经理的历史业绩怎么样", with `expected_outcome=manager_detail_blocked_fallback_manager`.

### 129. `composite_single` QCM `gate=always` is misleading when sub-intents may be blocked (MEDIUM)

Line 661 QCM lists `composite_single` with `gate=always`. But its sub-intents (e.g., `manager_detail`) may have `gate=blocked`. When a sub-intent is blocked, Registry filters it from `sub_intent_candidates`. The QCM's `always` gives no signal that implementers must check each sub-intent's gate individually.

Recommendation:

- Change `composite_single` gate to "取决于子 intent gate" or add a note: "子 intent 逐一按各自 gate 过滤；任一子 intent blocked 时不进入该子 intent 的候选".
