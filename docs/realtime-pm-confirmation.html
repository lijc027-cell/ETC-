ETF 实时查询 3 项确认表

这张表只聚焦 3 个问题：1. 意图如何映射到分类场景；2. 场景包含哪些需要查询的字段；3. 单位和展示口径怎么定。PM 只需要围绕这 3 件事给结论。

当前已确认的技术方向不在这张表里讨论：实时查询不走 Mongo；使用独立 realtime registry 和独立 embedding cards；后续代码实现会按 PM 确认后的这 3 张表落地。

一、这 3 个问题分别是什么

问题 1 PM 需要确认

用户问法应该归到哪个实时场景。例：盘口、折溢价、成交量额、实时行情概况。

问题 2 PM 需要确认

每个实时场景默认应该包含哪些查询字段。例：盘口要不要带买卖量。

问题 3 PM / 数据方需要确认

字段怎么展示给用户。例：成交额是不是要转亿元，premium 是价差还是比例。

目标 研发使用

PM 只要填完这 3 张表，研发就可以把这轮实时查询需求落到配置和代码里。

二、问题 1：意图 -> 分类场景

这张表只回答一件事：用户问某种问题时，应该归到哪个实时场景。后续代码里不会把这些关系写死，而是做成可替换配置。

用户问题类型 / 问法示例 建议分类场景 当前理解 PM 需要确认 PM 结论填写区
实时行情怎么样；现在什么情况；今天表现如何。 overview 理解为“实时行情概况”。 概况类是否就是一个独立场景。
现在多少钱；涨了吗；跌了多少；涨跌幅多少。 price_change 理解为“价格涨跌类”。 价格和涨跌幅是否独立于概况，还是并入概况。
成交额多少；成交量多少；交易活跃吗；现手多少。 trading 理解为“成交量额类”。 现手是否也归在成交类里。
IOPV 多少；实时净值估算；折溢价怎么样；溢价了吗。 valuation 理解为“IOPV / 折溢价类”。 IOPV 和折溢价是否是同一个场景。
盘口怎么样；买一卖一多少；买卖报价；挂单情况。 order_book 理解为“盘口类”。 盘口是否单独成一类。
内盘外盘多少；主动买卖情况；买盘卖盘活跃吗。 trade_flow 理解为“内外盘 / 成交方向类”。 这类是否作为第一版支持场景。
复合问法，例如“折溢价和成交额怎么样”。 多场景组合 理解为可同时命中多个场景。 第一版是否支持一个问题同时命中多个场景。

这张表只回答一件事：每个场景默认查哪些字段。后续可以直接改这张表，不需要改主流程代码。
